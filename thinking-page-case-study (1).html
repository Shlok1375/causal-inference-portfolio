When A/B Lift Lies:
Recovering Causal Effects
from a Non-Random Rollout

Shlok Sheth March 2026 PSM · DiD · Uplift Modeling GitHub ↗
TL;DR

A product team saw a 6.45% lift in conversions and nearly launched. The lift was real — but 85.7% of it was selection bias, not the feature. Using propensity score matching, difference-in-differences, and uplift modeling, I recovered a true ATE of 0.0092 and identified that the bottom 25% of users actually showed negative lift. The recommendation: targeted rollout to the top uplift quartile only.

The Setup

In an ideal world, every product feature gets a clean randomized experiment. In practice, rollouts are often staged, prioritized by user tier, or shaped by engineering constraints. When treatment assignment correlates with user quality, naive before/after or treated/control comparisons are biased — sometimes severely.

This project simulates exactly that scenario using a dataset structured like the Criteo Uplift Modeling Dataset (v2.1) — 100,000 users, 12 anonymized features, a non-random treatment flag, a pre-period visit signal, and a post-period conversion outcome.

⚠
The core problem: High-engagement users were more likely to receive the feature. Any metric difference between treated and control reflects both the feature's causal effect and the pre-existing quality gap between the groups. Naive lift conflates the two.

Step 1 — Diagnosing the Bias

Before applying any correction, I quantified imbalance using standardized mean differences (SMD) for each of the 12 features. An SMD above 0.10 indicates meaningful imbalance that will bias a naive comparison. 

# Standardized mean difference per feature
smd = ((treated_means - control_means) / pooled_std).abs()

Several features showed SMDs well above 0.10, confirming that the treated and control populations were systematically different before the feature was ever shown. The naive 6.45% lift is measuring mostly this pre-existing quality gap, not the feature's causal effect.

Method 1 — Propensity Score Matching

Propensity Score Matching (PSM) addresses selection bias by constructing a matched comparison group. Each treated user is paired with a control user who had the same estimated probability of being treated — effectively making the two groups statistically equivalent on observed features.

How it works

I estimated propensity scores using logistic regression on all 12 features, verified distributional overlap (common support), then ran 1:1 nearest-neighbor matching. After matching, the average SMD dropped from 0.088 to 0.014 — well below the 0.10 threshold on every feature. 

# Estimate propensity scores
lr = LogisticRegression(max_iter=1000)
lr.fit(X_scaled, treatment)
df['ps'] = lr.predict_proba(X_scaled)[:, 1]

# 1:1 nearest-neighbor matching
nn = NearestNeighbors(n_neighbors=1)
nn.fit(control_sample[['ps']])
distances, indices = nn.kneighbors(treated_sample[['ps']])

Method 2 — Difference-in-Differences

DiD uses time structure to control for unobserved baseline differences. Instead of comparing treated vs control directly, it compares the change in outcomes between periods. If both groups would have trended the same way without the feature (parallel trends), the difference in their changes is a valid causal estimate.

I used the visit column as a pre-period engagement proxy and conversion as the post-period outcome. The DiD estimate came out at 0.0189 — closer to the ground truth ATE of 0.0142 than the naive 0.0645. 

# DiD table
                   Treated   Control   Diff
Pre  (visit rate)  0.5962    0.5013    +0.0949
Post (conv rate)   0.3869    0.3730    +0.0139

DiD = (0.3869 - 0.5962) - (0.3730 - 0.5013)
    = 0.0189  ← causal estimate

Method 3 — Uplift Modeling

PSM and DiD both estimate a single population-average effect. But in product analytics, the actionable question is more granular: which users actually respond to the treatment, and which are unaffected or harmed?

I implemented a T-Learner (two-model approach) using Gradient Boosting — training separate models on treated and control users, then computing individual uplift as the difference in predicted conversion probabilities. 

# T-Learner: train separately on treated and control
model_t = GradientBoostingClassifier()
model_c = GradientBoostingClassifier()

model_t.fit(treated_data[features], treated_data['conversion'])
model_c.fit(control_data[features], control_data['conversion'])

# Individual uplift score
uplift = model_t.predict_proba(X)[:, 1] - model_c.predict_proba(X)[:, 1]

Recommendation

The naive 6.45% lift was almost entirely a measurement artifact. Both causal methods converge on a much smaller true effect — and uplift modeling reveals that this small average hides enormous heterogeneity across users.

  • Do Not Report the naive 6.45% lift to stakeholders or use it to justify a full launch. It overstates the true causal effect by 85.7%.
  • Correct Use PSM ATE of 0.0092 as the headline estimate, with the bootstrap 95% CI [0.0071, 0.0114] for statistical honesty.
  • Target Restrict rollout to users in the top uplift quartile. They show 5x the average lift — dramatically better ROI per treated user.
  • Avoid Rolling out to the bottom 25%. Observed lift is negative — treating these users actively hurts conversion.
→
Next step: Run a follow-up RCT restricted to high-uplift-score users. This validates the T-Learner's ITE estimates before a permanent launch decision and provides a clean causal estimate for a pre-specified population.

Limitations

PSM corrects only for observed confounders. Unmeasured variables — device type, geography, session depth — may still bias the estimate.

DiD relies on the parallel trends assumption. The visit rate is an imperfect proxy for pre-period behavior, and the assumption cannot be directly tested with a single pre-period.

Uplift model estimates were trained on observational data. Individual treatment effect predictions should be treated as directional signals, not causal facts, until confirmed via a controlled experiment.
